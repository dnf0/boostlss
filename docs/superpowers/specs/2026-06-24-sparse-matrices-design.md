# Sparse Matrix Support Design

## Overview
Currently, the `BoostLSS` engine stores its design matrix internally as a dense `ndarray::Array2<f64>`. This works well for many problems but causes massive memory overhead when dealing with high-dimensional datasets, particularly those generated by categorical one-hot encoding or many-knot splines.

This design document outlines the implementation of native Sparse Matrix support across the entire stack, from Python `scipy.sparse` inputs down to the Rust core engine.

## Core Architecture

### Rust: The `DesignMatrix` Enum
To avoid infecting the entire codebase with generics (`Dataset<M>`), we will use an Enum-based approach to encapsulate the design matrix. This allows us to keep a single `Dataset` type while supporting multiple underlying representations.

```rust
// In crates/boostlss/src/data.rs

#[derive(Clone, Debug, PartialEq)]
pub struct SparseMatrix {
    pub data: ndarray::Array1<f64>,
    pub indices: ndarray::Array1<usize>,
    pub indptr: ndarray::Array1<usize>,
    pub shape: (usize, usize),
}

#[derive(Clone, Debug, PartialEq)]
pub enum DesignMatrix {
    Dense(ndarray::Array2<f64>),
    Csr(SparseMatrix),
    Csc(SparseMatrix),
}
```

The `Dataset` struct will replace `design: Array2<f64>` with `design: DesignMatrix`.

### Feature Extraction via `get_column`
BoostLSS base-learners (like splines or linear models) evaluate one feature at a time. Therefore, we don't need complex sparse-matrix algebra; we just need fast column extraction. We will implement:

```rust
impl DesignMatrix {
    pub fn get_column(&self, col_idx: usize) -> ndarray::Array1<f64> {
        // ...
    }
}
```

- **Dense:** Returns a copy or view of the column.
- **CSC:** O(1) column access using `indptr[col_idx]` to `indptr[col_idx+1]`.
- **CSR:** O(N) extraction by iterating over rows and checking `indices` for the target `col_idx`.

## Python Bindings Architecture
To ensure zero-copy (or near zero-copy) transfers of sparse matrices from Python to Rust, we will extract the underlying NumPy arrays from `scipy.sparse` objects.

### `BoostLssModel.fit` and `predict` Logic
In the Python layer (e.g., inside a Python wrapper around the Rust `BoostLssModel` if we have one, or inside the PyO3 method handling arguments via `PyAny`), we handle the inputs:

1. **Check for `scipy.sparse`:**
   If `isinstance(X, scipy.sparse.spmatrix)`:
   - If `scipy.sparse.isspmatrix_csr(X)`, extract `X.data`, `X.indices`, `X.indptr` and call a new Rust method `fit_sparse(..., "csr")`.
   - If `scipy.sparse.isspmatrix_csc(X)`, extract arrays and call `fit_sparse(..., "csc")`.
   - Otherwise, convert it to CSC via `X.tocsc()` (since CSC is optimal for column extraction) and pass it down.
2. **Rust PyO3 Layer:**
   Add new PyO3 methods (or modify the existing ones to accept optional sparse components) that construct the `SparseMatrix` struct and wrap it in the `DesignMatrix::Csr` or `DesignMatrix::Csc` enum before creating the `Dataset`.

## Implementation Steps (For the Plan)
1. **Rust Core `data.rs`:** Define `SparseMatrix` and `DesignMatrix`. Update `Dataset` to use it.
2. **Rust Core Refactoring:** Fix all compilation errors resulting from `Dataset::design()` now returning a `DesignMatrix` instead of `Array2`. Update learners to use `.get_column(idx)`.
3. **Rust PyO3 Model (`model.rs`):** Update `BoostLssModel` to accept sparse data components. Create internal parsing logic to build the correct `Dataset`.
4. **Python Wrapper (if applicable) or Tests:** Write comprehensive Python integration tests passing `scipy.sparse.csr_matrix` and `scipy.sparse.csc_matrix` to `model.fit()` and `model.predict()`, ensuring results match the dense equivalents perfectly.
